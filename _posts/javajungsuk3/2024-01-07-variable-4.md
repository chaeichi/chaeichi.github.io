## 기본형(primitive type)
### 논리형 - boolean
- 논리형에는 `boolean` 한 가지만 존재
- `true`/`false` 중 하나를 저장할 수 있음
- **기본형은 false**
- 대답(yes/no), 스위치(on/off) 등의 논리구현에 주로 사용
- 두 가지의 값만 표현하면 되므로 1bit만으로 충분하지만 자바에서는 데이터를 다루는 최소 단위가 byte이므로 boolean의 크기는 `1byte`
- **자바는 대소문자가 구분되므로 TRUE와 true는 다름**

### 문자형 - char
- 문자형에는 `char` 한 가지만 존재
- 문자를 저장하기 위한 변수를 선언할 때 사용
- 단 하나의 문자만을 저장할 수 있음

```char
char ch = 'A' // 문자 'A'를 char 타입의 변수 ch에 저장
```

- 컴퓨터는 숫자밖에 모르기 때문에 **모든 데이터는 숫자로 변환하여 저장됨**
- 위 변수에도 문자가 아닌 `문자의 유니코드(정수)`가 저장됨
- 문자 리터럴 대신 문자의 유니코드(정수)를 직접 저장하는 것도 가능

```char
char ch = 'A' // 문자 'A'를 char 타입의 변수 ch에 저장
char ch = 65; // 문자의 유니코드(정수)를 직접 변수 ch에 저장
```

```char
char hch = 0xAC00; // char hch = '가';
char hch = '\uAC00'; // \u 다음 4자리의 16진수를 사용해 문자를 표현할 수 있음
```

- **문자의 유니코드**를 알고 싶으면 char형 변수에 저장된 값을 **정수형(int)으로 형변환(casting)**하면 됨

```char-to-int
int code = (int)ch;
```


```CharToCode.java
class CharToCode {
    public static void main(String[] args) {
        char ch = 'A'; // char ch = 65;
        int code = (int)ch; // ch에 저장된 값을 int 타입으로 변환하여 저장한다.
        System.out.printf("%c=%d(%#X)%n", ch, code, code);

        char hch = '가'; // char hch = 0xAC00;
        System.out.printf("%c=%d(%#X)%n", hch, (int)hch, (int)hch);
    }
}
```

```실행결과
A=65(0X41)
가=44032(0XAC00)
```

#### char타입의 표현형식
- 16비트로 표현할 수 있는 정수의 개수 : **2<sup>16</sup>(65536)개**
    - short 타입의 표현 범위 : **-2<sup>15</sup> ~ 2<sup>15</sup>-1(-32768 ~ 32767)**
    - char 타입의 표현 범위 : **0 ~ 2<sup>16</sup>-1(0 ~ 65535)**

- short 타입과 char 타입은 같은 2byte(16bit)
- char 타입은 음수를 나타낼 필요가 없기 때문에 `0 ~ 65535`의 범위를 가짐
- short 타입은 절반을 음수표현에 사용하기 때문에 `-32768 ~ 32767`의 범위로 가짐

```char-short
char ch = 'A'; // char ch = 65; 
short s = 65;
```

```char-short
System.out.println(ch); // A가 출력됨
System.out.println(s); // 65가 출력됨
```

- 변수 ch와 s에 `A`와 `65`를 저장하면 둘 다 **2진수로 똑같은 값이 저장됨**
- 그러나 변수의 값을 출력하면 결과는 다름
- `문자형`은 저장된 숫자에 해당하는 `유니코드 문자` 출력
- `정수형`은 변수에 저장된 값을 `10진수`로 해석하여 출력

#### 인코딩(encoding)과 디코딩(decoding)
- `인코딩(encoding)` : **문자를 코드로** 변환하는 것
- `디코딩(decoding)` : **코드를 문자로** 변환하는 것
- 문자를 저장할 때는 인코딩을 통해 숫자로 변환
- 저장된 문자(=숫자)를 읽어올 때는 디코딩을 통해 다시 문자로 변환
- 인코딩 방식을 알아야 디코딩이 가능

##### 아스키(ASCII)
- 미국정보교환표준코드(American Standard Code for Information Interchange)
- **128(2<sup>7</sup>)개의 문자 집합을 제공하는 7bit 부호**
    - 처음 32개의 문자는 인쇄와 전송 제어용으로 제어문자 출력 불가
    - 마지막 문자(DEL)를 제외한 33번째 이후의 문자들은 출력할 수 있는 문자
- 숫자 `0~9`, 영문자 `A~Z`, `a~z`가 연속적으로 배치

##### 확장 아스키(Extended ASCII)
- 아스키는 7bit이므로 1bit가 남음
- 확장 아스키는 **남는 1bit 공간을 활용해서 문자를 추가로 정의한 것**
- 확장 아스키에 추가된 128개의 문자는 여러 국가와 기업에서 서로의 필요에 따라 다르게 정의해서 사용
- 확장 아스키로 표현할 수 있는 문자의 개수로는 한글을 표현하기에 부족
- 두 개의 문자코드로 한글을 표현하는 방법을 고안
    - `조합형` : `초성`, `중성`, `종성`을 조합하는 방식
    - `완성형` : `확장 아스키의 일부 영역(162~254)`에 해당하는 `두 문자코드를 조합`하여 한글을 표현
    - 현재 조합형은 사용되지 않고 `완성형(KSC 5601)`에 없는 잘 안 쓰이는 8822글자를 추가한 `확장 완성형(CP 949)`이 사용 됨
    - 확장 완성형(CP 949)은 한글 윈도우에서 사용됨 
- 코드 페이지(code page, cp) : 확장 아스키의 256개 문자를 어떤 숫자로 변환할 것인지를 적어놓은 문자 코드표

##### 유니코드(Unicode)
- 인터넷이 발명되면서 서로 다른 지역의 다른 언어를 사용하는 컴퓨터간의 문서교환이 활발해짐
- 서로 다른 문자 인코딩을 사용하는 컴퓨터간의 문서교환에 어려움을 겪게 됨
- 이러한 어려움을 해소하고자 **전 세계의 모든 문자를 하나의 통합된 문자집합으로 표현**한 것이 유니코드
- 모든 문자를 2byte(2<sup>16</sup> = 65536)로 표현하려 했으나 부족해서 **21bit(약 200만 문자)**로 확장
    - 새로 추가된 문자들은 `보충문자(supplementary character)`라고 함
    - 보충문자는 `char` 타입이 아닌 `int` 타입 사용
- 유니코드 문자 집합(character set) : 유니코드에 포함하고자 하는 문자들의 집합
- 유니코드 인코딩(encoding) : 문자 집합에 번호를 붙인 것
    - UTF-8, UTF-16, UTF-32 등
    - `UTF-8` : 하나의 문자를 `1~4byte에 가변크기`로 표현
        - 영문과 숫자는 1byte, 한글은 3byte로 표현되기 때문에 `문서의 크기가 작음`
        - 문자의 크기가 가변적이므로 다루기 어려움
    - `UTF-16` : 모든 문자를 `2byte의 고정크기`로 표현
        - 모든 문자의 크기가 동일하여 문자를 다루기 편리
        - 1byte로 표현할 수 있는 영어와 숫자가 2byte로 표현되므로 `문서의 크기가 커짐`
    - 두 인코딩 모두 처음 128문자가 아스키와 동일
    - 자바에서는 주로 `UTF-16`을 사용
    - 인터넷에서는 전송속도가 유리하므로 **문서의 크기가 작은 UTF-8 인코딩으로 작성된 웹문서가 많음**

### 정수형 - byte, short, int, long
- byte(1byte), short(2byte), int(4byte), long(8byte)
- **기본 자료형은 int**

#### 정수형의 표현형식과 범위

![그림2-10 정수형의 표현형식](/assets/images/javajungsuk3/2-10.png)

- 모든 정수형은 `부호있는 정수`이므로 왼쪽의 첫 번째 비트는 `부호 비트(sign bit)`로 사용
- 나머지는 값을 표현하는데 사용
- 0으로 시작하는 2<sup>n-1</sup>개의 값을 양수 표현에 사용(0 포함)하고, 1로 시작하는 2<sup>n-1</sup>개의 값은 음수의 표현에 사용
- **n비트로 표현할 수 있는 정수의 개수 : 2<sup>n</sup>개(= 2<sup>n-1</sup>개 + 2<sup>n-1</sup>개)**
- **n비트로 표현할 수 있는 부호있는 정수의 범위 -2<sup>n-1</sup> ~ 2<sup>n-1</sup>-1**
    - 최대값에서 1을 빼는 이유는 범위에 `0`이 포함되기 때문
    - byte 타입(1byte = 8bit)의 변수에 저장할 수 있는 값의 범위는 -2<sup>7</sup> ~ 2<sup>7</sup>-1(-128 ~ 127)

#### 정수형의 선택 기준
- `JVM`의 `피연산자 스택(operand stack)`은 피연산자를 `4byte` 단위로 저장
- **크기가 4byte보다 작은 자료형(byte, short)의 값을 계산할 때는 4byte로 변환하여 연산이 수행됨**
- 정수형 변수를 선언할 때는 **int 타입을 사용하는 것이 효율적**
- int의 범위(약 ±20억)를 넘어서는 수를 다뤄야할 때는 long 사용
- byte나 short은 성능보다 저장공간을 절약하는 것이 중요할 때 사용
- long 타입의 범위를 벗어나는 값을 다룰 때는 실수형 타입이나 BigInteger 클래스를 사용

#### 정수형의 오버플로우
- `오버플로우(overflow)` : `해당 타입이 표현할 수 있는 값의 범위를 넘어서는 것`
- 만일 4bit 2진수의 최대값인 `1111`에 1을 더할 경우
    - `10000`이 되지만 4bit로는 4자리의 2진수만 저장할 수 있음
    - 5자리의 2진수 `10000`중에서 하위 4bit만 저장하여 `0000`이 됨
- 반대로 4bit 2진수의 최소값인 `0000`에 1을 뺄 경우, 최댓값인 `1111`이 됨
- 값을 무한히 증가시키거나 감소시켜도 `0000`과 `1111`의 범위를 계속 반복하게 됨
- 에러가 발생하는 것은 아니지만 예상했던 결과를 얻지 못하게 됨
- **최댓값 + 1 = 최솟값**
- **최솟값 - 1 = 최댓값**

#### 부호있는 정수의 오버플로우
- 부호없는 정수와 부호있는 정수는 표현범위, 즉 `최대값`과 `최소값`이 다르기 때문에 오버플로우 발생 시점이 다름
- `부호없는 정수`는 **2진수로 0000이 될 때** 오버플로우 발생
- `부호있는 정수`는 **부호 비트가 0에서 1이 될 때** 오버플로우 발생

![표2-16 부호없는 정수와 부호있는 정수의 오버플로우](/assets/images/javajungsuk3/2-16-table.png)

```OverflowEx.java
class OverflowEx {
    public static void main(String[] args) {
        short sMin = -32768;
        short sMax = 32767;
        char cMin = 0;
        char cMax = 65535;

        System.out.println("sMin    = " + sMin);
        System.out.println("sMin-1  = " + (short)(sMin-1));
        System.out.println("sMax    = " + sMax);
        System.out.println("sMax+1  = " + (short)(sMax+1));
        System.out.println("cMin    = " + (int)cMin);
        System.out.println("cMin-1  = " + (int)--cMin);
        System.out.println("cMax    = " + (int)cMax);
        System.out.println("cMax+1  = " + (int)++cMin);
    }
}
```

```실행결과
sMin    = -32768
sMin-1  = 32767
sMax    = 32767
sMax+1  = -32768
cMin    = 0
cMin-1  = 65535
cMax    = 65535
cMax+1  = 0
```

### 실수형 - float, double
- float(4byte), double(8byte)
- **기본 자료형은 double**
- 실수형은 `소수점수`도 표현해야 하므로 `얼마나 큰 값을 표현할 수 있는가`뿐만 아니라 `얼마나 0에 가깝게 표현할 수 있는가`도 중요하다.
- 정수형과 달리 실수형에서는 오버플로우가 발생하면 변수의 값은 `무한대(infinity)`가 됨
- 정수형에는 없는 `언더플로우(underflow)`가 존재. `실수형으로 표현할 수 없는 아주 작은 값, 즉 양의 최소값보다 작은 값이 되는 경우`
- 이 때 변수의 값은 `0`이 됨

#### 실수형의 범위와 정밀도

![그림2-12 float타입으로 표현할 수 있는 값의 범위](/assets/images/javajungsuk3/2-12.png)
- float 타입의 값의 범위는 **-3.4 x 10<sup>38</sup> ~ 3.4 x 10<sup>38</sup>**이지만 **-1.4 x 10<sup>45</sup> ~ 1.4 x 10<sup>45</sup> 범위(0은 제외)**의 값은 표현할 수 없음

#### 실수형의 표현형식
- 정수형과 실수형은 값을 저장하는 형식이 다름

![그림2-13 int타입과 float타입의 표현형식](/assets/images/javajungsuk3/2-13.png)

- int 타입과 같은 정수형은 `부호`와 `값`. 두 부분으로 이루어져 있음
- float 타입과 같은 실수형 `부호(S)`와 `지수(E)`, `가수(M)`. 세 부분으로 이루어져 있음
- 실수형은 `부동소수점(floating-point)`의 형태로 저장
    - **2의 제곱을 곱한 형태(±M X 2<sup>E</sup>)**로 저장하기 때문에 큰 범위의 값을 저장하는 것이 가능

1. 부호(sign bit)
    - 0이면 양수, 1이면 음수
    - 정수형과 달리 2의 보수법을 사용하지 않기 때문에 **부호비트만 변경하는 것으로 양의 실수를 음의 실수로 바꿀 수 있음**
2. 지수(exponent)
    - 부호있는 정수
    - float는 `-127 ~ 128`의 범위를 가짐
    - 이 중 `-127`과 `128`은 `숫자 아님(NaN, Not a Number)`이나 `양의 무한대(POSITIVE_INFINITY)`, `음의 무한대(NEGATIVE INFINITY)`와 같은 특별한 값을 위해 예약되어 있음
    - 실제 사용 가능한 지수의 범위는 `-126 ~ 127`
    - 지수의 최대값이 127이므로 float 타입으로 표현할 수 있는 최대값은 2<sup>127</sup>, 10진수로 약 10<sup>38</sup>
    - 그러나 float의 최소값은 가수의 마지막 자리가 2<sup>-23</sup>이므로 지수의 최소값보다 2<sup>-23</sup>배나 더 작은 값인 약 10<sup>-45</sup>
    - double은 `-1023 ~ 1024`의 범위를 가짐
3. 가수(mantissa)
    - 실제 값을 저장하는 부분
    - float는 2진수 23자리, 10진수로 약 7자리 저장 가능
    - 즉, 정밀도가 `7자리`로 **7자리의 10진수를 오차없이 저장할 수 있음**
        - 1234.567 = 1.234567 x 10<sup>3</sup>
        - 0.00001234567 = 1.234567 x 10<sup>-5</sup>
        - 1234567000 = 1.234567 x 10<sup>9</sup>
        - 위 세 값은 float 타입의 변수에 저장했을 때 오차없이 저장 가능
    - double은 2진수 52자리, 10진수로 약 15자리 저장 가능
    - 즉, 정밀도가 `15자리`로 **15자리의 10진수를 오차없이 저장할 수 있음**
    - double 타입이 **float 타입보다 훨씬 더 정밀하게 값을 표현할 수 있음**
    - 실수형 값을 저장할 때, float 타입이 아닌 double 타입의 변수를 사용하는 것은 `값의 범위` 때문이 아닌 `보다 높은 정밀도` 때문
    - 연산속도의 향상이나 메모리 절약을 위해서는 float 선택

![그림2-14 float과 double의 표현형식](/assets/images/javajungsuk3/2-14.png)

- 정수형과 달리 실수형은 오차가 발생할 수 있다는 단점 존재
- 표현할 수 있는 값의 범위뿐만 아니라 `정밀도(precision)`도 중요한 요소

```FloatEx1.java
class FloatEx1 {
    public static void main(String[] args){
       float f = 9.12345678901234567890f;
       float f2 = 1.2345678901234567890f;
       double d = 9.12345678901234567890d;

        System.out.printf("         123456789012345678901234%n");
        System.out.printf("f    :     %f%n", f); // 소수점 이하 6째자리까지 출력
        System.out.printf("f    :   %24.20f%n", f);
        System.out.printf("f2   :   %24.20f%n", f2);
        System.out.printf("d    :   %24.20f%n", d);
    }
}
```

#### 부동소수점 오차
- 실수 중에는 `파이(𝝿)`와 같은 `무한소수`가 존재
- 10진수로는 `유한소수`이더라도 2진수로 변환하면 `무한소수`가 되는 경우가 존재
- 2진수로 유한소수라도 가수를 저장할 수 있는 자리수는 한정되어 있음
- 저장되지 못하고 버려지는 값이 생기면 오차가 발생
- `정규화` : 2진수로 변환된 실수를 저장할 때 먼저 **1.xxx X 2<sup>n</sup>**의 형태로 변환하는 과정
- 정규화된 2진 실수는 항상 **1.**으로 시작하기 때문에 이를 제외한 23자리의 2진수가 가수로 저장되고 그 이후는 잘려나감
- `기저법` : 2의 보수법처럼 `부호있는 정수`를 저장하는 방법. 저장할 때 `특정값(기저)`을 더했다가 읽어올 때는 다시 뺌
- 지수는 `기저법`으로 저장되기 때문에 지수인 3에 기저인 127을 더한 130이 2진수로 변환되어 저장
- 이 때 잘려나간 값들에 의해 발생할 수 있는 최대오차는 약 **2<sup>-23</sup>**(가수의 마지막 비트의 단위)
- **2<sup>-23</sup>**은 약 **10<sup>-7</sup>**이므로 float의 정밀도를 `7자리`라 하는 것

```FloatEx2.java
class FloatEx2 {
    public static void main(String[] args) {
        float f = 9.1234567f;
        int i = Float.floatToIntBits(f);
        
        System.out.printf("%f%n", f);
        System.out.printf("%X%n", i); // 16진수로 출력
    }
}
```

```실행결과
9.123457
4111F9AE
```

- `floatToIntBits` : `부동소수점 값의 표현 반환`. float 타입의 값을 int 타입의 값으로 해석해서 반환
- 9.1234567은 2진수로 변환했을 때 잘려나간 첫 번째 자리의 값이 `1`이라 반올림되어 0x4111F9AD에서 0x4111F9AE가 됨
- **2진수는 1이 절반이기 때문에 1이면 반올림**

## 🚩 잊지않기
- 16비트로 표현할 수 있는 정수의 개수 : **2<sup>16</sup>(65536)개**
    - short 타입은 절반을 음수표현에 사용하기 때문에 **-2<sup>15</sup> ~ 2<sup>15</sup>-1(-32768 ~ 32767)**의 범위로 가짐
    - char 타입은 음수를 나타낼 필요가 없기 때문에 **0 ~ 2<sup>16</sup>-1(0 ~ 65535)**의 범위를 가짐
- **유니코드 인코딩(encoding)**
    - `UTF-8` : 하나의 문자를 `1~4byte에 가변크기`로 표현
        - 영문과 숫자는 1byte, 한글은 3byte로 표현되기 때문에 `문서의 크기가 작음`
        - 문자의 크기가 가변적이므로 다루기 어려움
    - `UTF-16` : 모든 문자를 `2byte의 고정크기`로 표현
        - 모든 문자의 크기가 동일하여 문자를 다루기 편리
        - 1byte로 표현할 수 있는 영어와 숫자가 2byte로 표현되므로 `문서의 크기가 커짐`
    - 인터넷에서는 전송속도가 유리하므로 **문서의 크기가 작은 UTF-8 인코딩으로 작성된 웹문서가 많음**
- **정수형**
    - `JVM`의 `피연산자 스택(operand stack)`은 피연산자를 `4byte` 단위로 저장하기 때문에 크기가 4byte보다 작은 자료형(byte, short)의 값을 계산할 때는 4byte로 변환하여 연산이 수행됨
    - `오버플로우(overflow)`는 해당 타입이 표현할 수 있는 값의 범위를 넘어서는 것을 의미
        - **최댓값 + 1 = 최솟값**
        - **최솟값 - 1 = 최댓값**
- **실수형**
    - 정수형에 없는 `언더플로우(underflow)`가 존재. `실수형으로 표현할 수 없는 아주 작은 값, 즉 양의 최소값보다 작은 값이 되는 경우`*(이 때 변수의 값은 `0`이 됨)*
    - 정수형과 달리 2의 보수법을 사용하지 않기 때문에 **부호비트만 변경하는 것으로 양의 실수를 음의 실수로 바꿀 수 있음**
    - 정수형과 달리 오차가 발생할 수 있다는 단점이 존재하므로 표현할 수 있는 값의 범위뿐만 아니라 `정밀도(precision)`도 중요한 요소
